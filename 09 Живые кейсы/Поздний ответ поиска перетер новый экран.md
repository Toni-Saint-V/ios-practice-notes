# Поздний ответ поиска перетер новый экран

> **Коротко:** Классика: пользователь уже ввел новый запрос, а старый ответ приехал позже и победил.

## Ситуация
Экран поиска отелей. Пользователь быстро набрал `par`, потом `paris`. Запрос `paris` вернулся первым, UI показал нормальные результаты. Через секунду вернулся старый `par` и перетер экран.

На глаз это выглядит как «поиск иногда странный». В коде обычно видно другое: состояние обновляется по факту любого success, без проверки, актуален ли этот success.

## Что насторожило
- `Task` отменяется, но после `await` нет проверки актуальности.
- State хранит только `[Hotel]`, без query/request id.
- Тесты покрывают только happy path.

## Нормальная защита
```swift
let requestID = UUID()
latestRequestID = requestID

let result = try await service.search(query)

guard latestRequestID == requestID else {
    return
}

state = .content(query: query, hotels: result)
```

Отмена полезна, но request id все равно нужен. Особенно если внутри есть legacy callback API, кеш, retry или слой, который не идеально уважает cancellation.

## Проверка
- Первый запрос зависает.
- Второй запрос завершается.
- Первый запрос завершается позже.
- В UI остается второй результат.
