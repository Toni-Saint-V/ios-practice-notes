# База Swift и UIKit

> **Коротко:** Базовые темы здесь нужны не для повторения синтаксиса. Они закрывают места, где простые механизмы Swift и UIKit неожиданно ломают состояние, память, layout и тесты.

## Optional и состояние экрана
`Optional` удобен, пока `nil` означает ровно одну вещь. Проблемы начинаются, когда один `nil` одновременно значит «еще не загрузили», «нет данных», «ошибка», «нет доступа» и «данные очищены после logout».

Лучше разделять состояния явно:

```swift
enum ProfileState: Equatable {
    case idle
    case loading
    case content(Profile)
    case empty
    case forbidden
    case error(message: String)
}
```

Редкая поломка: после logout профиль становится `nil`, экран считает это empty state и показывает кнопку «создать профиль», хотя правильное состояние — закрытая сессия.

## ARC, замыкания и жизненный цикл
Утечки чаще появляются не потому, что кто-то не знает `weak self`, а потому что неясно, кто владеет задачей.

```swift
final class PaymentViewModel {
    private var task: Task<Void, Never>?

    func start() {
        task = Task { [weak self] in
            guard let self else { return }
            await self.loadPayment()
        }
    }

    deinit {
        task?.cancel()
    }
}
```

Проверка на ревью: если объект запускает долгую работу, должно быть понятно, когда эта работа отменяется.

## Property Wrapper без магии
`@State`, `@StateObject`, `@ObservedObject`, `@EnvironmentObject` — это не украшения. Они отвечают на вопрос владения.

- `@State` — локальное значение внутри view.
- `@StateObject` — view создает и держит объект.
- `@ObservedObject` — объект пришел снаружи.
- `@EnvironmentObject` — зависимость общая, но ее легко сделать слишком неявной.

Редкая поломка: `ObservableObject` создали внутри `body`, родитель перерисовался, объект пересоздался, экран потерял состояние.

## Auto Layout и SwiftUI layout
Layout ломается не на обычном тексте, а на длинной локализации, Dynamic Type, split view, клавиатуре и состоянии ошибки.

Что проверять:

- длинный текст кнопки;
- XXL Dynamic Type;
- right-to-left, если продукт это поддерживает;
- ошибка под полем;
- loading вместо текста;
- пустой список и список с реальными данными.

## UIKit lifecycle
`viewDidLoad` не значит «экран сейчас виден». `viewWillAppear` не значит «надо заново грузить все данные». `viewDidDisappear` не всегда значит «экран умер».

Для UIKit-экрана важно отдельно решить:

- где создается состояние;
- что перезапускается при возвращении на экран;
- что отменяется при уходе;
- как экран переживает deep link, logout и смену root flow.

## Паттерны
Паттерн полезен, если он уменьшает конкретную боль.

- Strategy хороша для смены алгоритма: retry policy, sorting, formatting.
- Delegate хорош для событий от владельца наружу.
- Decorator хорош для добавления поведения: логирование, кеш, метрики.
- Proxy хорош для контроля доступа или lazy-загрузки.

Плохой сигнал: паттерн появился раньше проблемы, а код стал длиннее без понятной границы ответственности.

## Самопроверка
- `nil` в модели означает одно состояние или несколько разных?
  Ответ: если несколько, нужен `enum`.
- Понятно ли, кто отменяет async-задачу?
  Ответ: владелец задачи должен отменять ее при завершении своего жизненного цикла.
- Property wrapper отражает владение?
  Ответ: если объект создается здесь — `@StateObject`; если пришел снаружи — `@ObservedObject`.
- Layout проверен на длинном тексте и Dynamic Type?
  Ответ: без этого UI проверен только на самый удобный случай.
